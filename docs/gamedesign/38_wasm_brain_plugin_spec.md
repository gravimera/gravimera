# Wasm Brain Plugin Spec (Unit AI Modules)

_(Spec document; product intent lives in `docs/gamedesign/06_brains_and_ai.md`.)_

This document defines a **final target** interface for “AI plugins” that can fully control a unit via **sandboxed WebAssembly**. A Wasm brain is an optional advanced brain type alongside:

- external agents driving units via the HTTP API, and
- embedded behavior graphs (`docs/gamedesign/18_behavior_graph_spec.md`).

The core idea is: **sense → decide → act**, where the host simulation is always authoritative.

## Design Goals

1) **Full unit control**: movement, abilities/actions, interaction, and speech.
2) **Host-authoritative**: AI emits *requests*, the simulation validates and executes.
3) **Deterministic**: same realm state + same tick stream => same outputs.
4) **Safe to host**: strict sandbox, budgets, and explicit capabilities.
5) **Scales to many units**: bounded inputs/outputs, AI LOD, and parallelizable execution.
6) **Debuggable**: clear command outcomes, traceability, and replay tooling.

## Non-Goals

- Allow arbitrary filesystem/network access from user code.
- Let a brain mutate the world state directly (no “write access to ECS”).
- Provide unlimited perception queries (unbounded sensing becomes an attack/perf vector).

## Execution Model

Each unit may have one attached Wasm brain instance.

### Lifecycle Hooks

- `init(config)`: called once when the brain is attached/spawned (optional).
- `tick(input) -> output`: called when the brain is scheduled to run.
- `on_despawn(reason)`: optional, for cleanup.

Hosts may also support `reset()` for hot-reload or recovery after failures.

### Tick Frequency

The host decides when to call `tick()`:

- **Baseline**: once per simulation step (`FixedUpdate`), e.g. 30–60Hz.
- **Event-driven**: idle brains can “sleep” and be woken early by events.
- **AI LOD**: the host can tick less important units at lower rates (e.g. 2–10Hz).

`tick()` MUST be treated as a synchronous call (no yielding); the host enforces budgets via fuel/timeouts.

## Contract Boundary: Requests, Not Authority

Brains do not directly change simulation state. They emit **intents/commands** such as:

- movement/aim control,
- “use ability X on target Y”,
- “say text to nearby units / channel”,
- “interact with entity”.

The host validates each request against rules (range, cooldowns, permissions, collision, line-of-sight, etc). Validation outcomes are reported back to the brain via command result events.

## Data Interfaces (Sense → Decide → Act)

This spec assumes the **Wasm Component Model + WIT** is used for a stable, typed ABI.

### Input: `TickInput`

`TickInput` is a bounded, deterministic snapshot containing:

- **Self state**: transform/pose, velocity, health/stamina, statuses, inventory summary, cooldowns, current target, team/species/faction tags.
- **Perception snapshot** (bounded):
  - nearby entities (stable ordering), each with: `entity_id`, kind/type, tags (species/faction), relative pose/vel, threat/relationship hints.
  - nearby hazards/projectiles/areas of effect (optional module data).
  - navigation hints (optional): local walkability/heightmap sample, current path segment id, etc.
- **Events since last tick** (see “Event Model” below).
- **Time**: fixed `dt_ms`, `tick_index`, realm time (if enabled).
- **Deterministic randomness**: RNG seed or a stream position derived from `(realm_seed, unit_id, brain_id, tick_index)`.
- **Capabilities**: an explicit capability set granted to this brain instance (read-only).

All lists must have explicit caps (`max_nearby_entities`, `max_events_per_tick`, etc). When caps are exceeded, the host reports truncation explicitly.

### Output: `TickOutput`

`TickOutput` is a bounded set of requested actions:

- **Motor control** (recomputed every tick; last output does not imply authority):
  - `set_move(vec2)` / `set_throttle(f32)` / `set_look(yaw,pitch)` (game-dependent), or
  - `move_to(position)` if pathfinding is host-side.
- **Discrete commands**:
  - `use_ability(ability_id, target_id|position)`
  - `interact(entity_id)`
  - `equip(item_id)` / `drop(item_id)`
  - `set_target(entity_id)` / `clear_target`
- **Speech/social**:
  - `say(channel, text, target_id?)`
  - `emote(kind)`
- **Scheduling hint**:
  - `sleep_for_ticks(n)` (host may still wake early on events).

Hosts should treat `TickOutput` as “desired actions” and apply them through a single authoritative command executor.

## Event Model

Events are generated by the **host simulation** and delivered to the brain in `TickInput.events`.

### Event Delivery Rules

- Each unit has a bounded **event inbox** (ring buffer).
- `tick()` receives “events since last tick”, in a deterministic order.
- If overflow occurs, the host emits a synthetic event such as:
  - `EventsDropped { count }`, or
  - `PerceptionReset { snapshot }` (for perception-derived events).

### SeenEnter / SeenExit (Perception Edges)

`SeenEnter`/`SeenExit` are **edge events**: they are emitted only when a target transitions:

- not-visible → visible (`SeenEnter`)
- visible → not-visible (`SeenExit`)

They are detected at the rate the **perception system** runs:

- if perception updates every sim step, edge detection is ~per-tick resolution;
- if perception updates at a slower sensor rate (e.g. 10Hz), events can be delayed by up to the sensor period.

### Event Producers (Examples)

- **Perception system**: `SeenEnter`, `SeenExit`, `TargetLost`, `ThreatUpdated`.
- **Speech propagation system**: `HeardSpeech { speaker_id, text, channel, rel_pos, confidence }`.
- **Combat/rules systems**: `Damaged`, `Healed`, `Died`, `StatusApplied`, `AbilityReady`.
- **Interaction system**: `InteractRequested`, `TradeOffered`.
- **Command executor**: `CommandAccepted`, `CommandRejected { reason }`, `CommandResult { success, details }`.

Each event type should be rate-limited and bounded to prevent denial-of-service via spam (especially speech).

## Perception / Visibility System (Gameplay Sensing)

Bevy’s built-in `Visibility` is for rendering; unit perception is a **gameplay system** that must be implemented explicitly.

### Baseline Algorithm (Deterministic, Bounded)

For each sensor update:

1) Broadphase query nearby candidates from a spatial index (uniform grid / spatial hash).
2) Filter by deterministic rules: range, FOV, team/species masks, stealth tags.
3) Optionally apply line-of-sight checks (raycast) with strict budgets.
4) Produce the current `visible_set` (stable ordering).
5) Diff against `previous_visible_set` to emit `SeenEnter`/`SeenExit`.

### Performance Requirements

To support many units:

- Avoid `O(n²)` by using a spatial index.
- Run perception at a **sensor rate** (5–20Hz) instead of every tick when possible.
- **Stagger** sensor updates across ticks (bucket units by `unit_id % N`) to smooth spikes.
- Apply **AI LOD**: far/idle/offscreen units get slower sensors, smaller radii, fewer LOS checks, or full sleep.
- Do LOS checks only for the nearest `K` candidates, the current target, or newly-entered candidates; cache results for a small TTL.

## Host Threading Model

Wasm calls are synchronous. Parallelism is achieved by running many brains concurrently on the host.

### Single-Thread (Simple)

- In the sim step, for each unit: build `TickInput`, call `tick()`, apply `TickOutput`.
- Deterministic and easy to debug; limited throughput.

### Parallel Snapshot → Run → Apply (Scalable)

1) **Snapshot** (main thread): build read-only `TickInput` per unit and drain each inbox.
2) **Run** (worker pool): call `tick()` for many units in parallel (no ECS/world access).
3) **Apply** (main thread): collect outputs, sort by stable key (e.g. unit id), and execute via a command executor.

Notes:

- Don’t share a single Wasm instance/store across threads; instead use per-unit instances or shard instances by worker.
- Sorting outputs before applying improves determinism when parallelizing.

## Safety, Governance, and Fairness

### Sandbox Rules

- No filesystem/network access from Wasm brains.
- No direct clocks; only host-provided deterministic time.
- Only whitelisted host imports, each with explicit budgets.

### Budgets and Backpressure

Per brain tick:

- CPU budget (fuel/instruction limit and/or timeout)
- max memory
- max host calls (per import and total)
- max output commands (and max speech bytes)

On repeated traps/timeouts:

- emit `BrainFaulted`,
- downgrade LOD or disable brain per realm policy,
- fall back to a safe built-in brain (optional).

### Capability Gating

Brains declare required capabilities (e.g. `combat`, `talk`, `trade`, `pathfind`). The realm/host grants a subset. Requests outside granted capabilities must be rejected with explicit reasons (and surfaced to the brain via `CommandRejected`).

### Provenance and Trust

Hosts may require:

- pre-compilation in a sandboxed build service,
- module signing (content hash + signature),
- metadata: ABI version, declared imports, declared capabilities.

## Persistence and Versioning

### Persistent Brain Memory

If brains need memory beyond a blackboard:

- provide a small bounded key/value store (typed) managed by the host,
- enforce per-unit quotas,
- serialize deterministically as part of realm saves.

### ABI Versioning

The Wasm brain interface MUST declare an `abi_version` (and optionally feature flags). Hosts may reject mismatched versions. (Early-stage development can bump versions freely; compatibility is not required.)

## Tooling and Observability (Required)

- Deterministic “brain runner” harness for local testing.
- Reproducible replay: record `(TickInput, TickOutput, command results)` for a unit and replay offline.
- Structured logging/tracing (rate-limited) tied to unit id and brain id.
- Per-brain metrics: tick time, fuel used, host calls, event drops, command rejects.
